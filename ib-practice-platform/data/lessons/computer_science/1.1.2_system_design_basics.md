## System Design Basics

## Overview

This lesson introduces the core principles of system design, focusing on decomposition, modularity, and the concept of a system as a collection of interacting components. Students will explore how systems are built and how different layers contribute to functionality, setting a foundation for understanding more complex system architectures.

## Key Concepts

*   **System Decomposition:** Breaking down a complex system into smaller, manageable components.
*   **Modularity:** Designing components that are self-contained, reusable, and have well-defined interfaces.
*   **System Layers:** Understanding the hierarchical structure of a system, where each layer performs a specific function.
*   **Interface Design:** Creating clear and concise interfaces between components to facilitate interaction.

### 1. What is a System?

> **Important:** A **system** is an interconnected set of components that work together to achieve a specific goal. This could be anything from a simple calculator to a complex operating system. Systems aren't just hardware; they encompass software, data, and the people who interact with them.

Systems can be categorized based on their complexity:

*   **Simple Systems:** These have few components and clear, linear processes. (e.g., a vending machine).
*   **Complex Systems:** These have many interconnected components with emergent behavior – meaning the overall behavior isn't simply the sum of the individual parts. (e.g., the Internet).

**Example:** Consider a simple online shopping system. It’s comprised of:

*   **User Interface (UI):** The website or app the user interacts with.
*   **Application Server:** Processes user requests (e.g., adding items to a cart).
*   **Database:** Stores product information, user accounts, and order details.
*   **Payment Gateway:** Handles secure transactions.

Each of these components has a specific function, and they interact to facilitate the entire shopping process.

### 2. System Decomposition – Breaking it Down

> **Exam Tip:**  The IB assesses your ability to analyze and describe how systems are designed.  Command words like "describe" and "explain" will be frequently used.

System decomposition is the process of breaking down a large, complex system into smaller, more manageable **subsystems** or **modules**. This makes the system easier to understand, develop, test, and maintain.  It’s fundamentally about dividing and conquering.

**Worked Example 1:**

Let’s decompose a music streaming application:

*   **High-Level System:** Music Streaming Application
*   **Decomposed Subsystems:**
    *   **User Authentication:** Handles user login and registration.
    *   **Music Catalog Management:**  Stores and retrieves information about songs and artists.
    *   **Playback Engine:**  Handles audio streaming and control.
    *   **Recommendation Engine:** Suggests music based on user preferences.
    *   **Payment Processing:** Handles subscription payments.

This decomposition allows developers to work on individual subsystems concurrently, reducing development time and complexity.

### 3. Modularity – Building Blocks

> **Note:** Modularity promotes code reusability and simplifies maintenance. Well-designed modules have clearly defined interfaces.

**Modularity** is a design principle where a system is constructed from independent, self-contained **modules** (or **components**). Each module performs a specific task and interacts with other modules through well-defined **interfaces**.

**Key characteristics of modular systems:**

*   **Loose Coupling:** Modules have minimal dependencies on each other. Changes in one module have less impact on others.
*   **High Cohesion:** Each module focuses on a single, well-defined task.
*   **Reusability:** Modules can be used in multiple parts of the system or even in different systems.

**Example:** In object-oriented programming, a **class** represents a module.  A `Car` class might contain methods for starting the engine, accelerating, and braking. These methods are encapsulated within the class and can be used by other parts of the application.

### 4. System Layers – A Hierarchical Approach

> **Exam Tip:** Understand the concept of layering to effectively analyze system architecture. Diagrams are crucial for illustrating layered systems.

Most systems are organized into **layers**, which represent different levels of abstraction. Each layer builds upon the services provided by the layer below it.  This creates a hierarchical structure, simplifying design and maintenance.

**Common System Layers:**

*   **Presentation Layer:**  The user interface (UI) – what the user sees and interacts with.
*   **Application Layer:**  Handles business logic and processes user requests.
*   **Business Logic Layer:**  Contains the core logic of the system.
*   **Data Access Layer:**  Provides an interface for accessing and manipulating data.
*   **Database Layer:**  Stores and manages the system's data.

### 5. Interface Design – Communication Protocols

> **Important:** A well-designed interface minimizes ambiguity and promotes seamless communication between modules.

An **interface** defines how components interact.  It specifies the methods, data structures, and protocols that are used to exchange information. A good interface is:

*   **Clear and Concise:** Easy to understand and use.
*   **Well-Documented:** Provides clear instructions and examples.
*   **Stable:** Changes to the interface should be minimized to avoid breaking compatibility.

**Example:**  A function that takes a list of numbers as input and returns their average needs a clearly defined input type (list of numbers) and output type (the average).

### 6.  System Design Considerations

*   **Scalability:** The ability of the system to handle increasing loads (users, data, requests).
*   **Reliability:** The ability of the system to function correctly under all circumstances.
*   **Maintainability:** The ease with which the system can be modified and updated.
*   **Security:** Protecting the system and its data from unauthorized access and attacks.

**Worked Example 2:**

Imagine designing a social media platform.  You'd need to consider scalability to handle millions of users and data.  Reliability is crucial – the platform should remain available even during peak usage. Security is paramount – protecting user data from breaches.  These considerations heavily influence the system architecture.
